# Agent Loop Operating Model

이 문서는 EnglishProject에서 Codex가 작업을 진행할 때 따르는 반복 구조를 정의합니다.

## 1. Ralph 루프

- 역할: 최상위 반복 드라이버
- 상태 위치: `C:/Users/woung/Desktop/EnglishProject/.codex/ralph-loop/`
- 판단 기준:
  - 결과가 목표와 다르면 `REVISE`
  - 목표와 검증을 통과하면 `SHIP`
  - 로컬 정보나 권한 문제로 진행할 수 없으면 `BLOCKED`
- 적용 방식: 큰 작업이 끝날 때마다 목표 대비 결과를 확인하고 다음 반복 여부를 판단한다.

## 2. Codex 오케스트레이터 루프

- 역할: 중간 관리자
- 흐름:
  1. 사용자 요청을 목표와 제약으로 정리한다.
  2. 작업을 작은 단위로 나눈다.
  3. 구현, 명세 검토, 코드 품질 검토, 검증을 분리한다.
  4. 결과가 부족하면 다시 작업 단위로 되돌린다.
- 적용 방식: 사용자가 작업 요청을 하면 이 루프가 먼저 돈다.

## 3. Superpowers 작업 루프

- 역할: 실제 구현 방식
- 흐름:
  1. 작업 분해
  2. 구현 에이전트 관점의 변경
  3. 명세 리뷰
  4. 코드 품질 리뷰
  5. 수정 반복
- 적용 기준: 큰 버그 수정, 기능 구현, 제품 흐름 변경, 평가 로직 변경에 사용한다.

## 4. Verification 루프

- 역할: 완료 주장 전 검증 게이트
- 원칙:
  - 테스트, 타입 체크, 빌드, 콘텐츠 검증 등 필요한 명령을 실제로 실행한다.
  - 실행 결과를 확인하기 전에는 완료됐다고 말하지 않는다.
  - APK가 필요한 변경이면 로컬 Gradle 빌드로 생성한다.
- 현재 기본 검증:
  - `npm.cmd test` in `app`
  - `npm.cmd run typecheck` in `app`
  - `npm.cmd run content:validate` in `app`
  - 필요 시 `npm.cmd run build:android:release` in `app`

## 5. Review 루프

- 역할: 결과물이 제품 목표와 맞는지 독립 검토
- 검토 관점:
  - 테스트 통과 여부
  - 현재 언어 학습 목표와의 정합성
  - 사용자가 실제로 겪은 문제를 해결했는지
  - 오답/정답 판정, 반복 출제, 회화 흐름이 학습 경험을 해치지 않는지
- 적용 방식: 기능이 끝나면 “테스트는 통과했지만 제품적으로 맞는가”를 별도로 확인한다.

## 6. Compound 루프

- 역할: 같은 실수를 반복하지 않도록 교훈 저장
- 저장 위치: `docs/solutions/`
- 적용 기준:
  - 문제 해결 후 새 교훈이 생겼을 때
  - 같은 유형의 실수가 반복될 가능성이 있을 때
  - 평가 로직, 문제팩, APK 빌드, 로컬 상태 저장처럼 재사용 가치가 있을 때
- 적용 방식: 매번 자동으로 남기지 않고, 리뷰 후 남길 가치가 있는 경우에만 기록한다.

## 운영 규칙

- 버그 수정은 재현 테스트를 먼저 만든다.
- 완료 전에는 Verification 루프를 반드시 통과한다.
- 수정이 완료되면 바로 커밋해서 되돌릴 수 있게 한다.
- 릴리즈 APK를 만들 때는 원격 빌드가 아니라 로컬 Android 빌드 도구를 우선 사용한다.
- 새 교훈이 있으면 Compound 루프를 통해 `docs/solutions/`에 남긴다.
